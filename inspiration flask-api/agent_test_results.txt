  14. CAPABILITY SUMMARY (shown to LLM)

[ANALYSIS]
  - run_ai_suggestions: Run AI gap analysis to identify missing risks and controls in the RCM. Generates suggestions based on industry best practices. Results are cached — call merge_suggestions afterwards to apply.
  - run_control_assessment: Run OnGround Check: validate RCM controls against policy and SOP documents. Checks whether each control is documented in the provided policies/SOPs and identifies gaps. Provide at least one policy or SOP file path.
  - run_deduplication: Analyze the RCM for semantically duplicate risks and controls. Uses AI to compare entries within each process group. Results are cached — call remove_duplicates afterwards to apply.
  - preview_tod_attributes: Generate and preview TOD testing attributes for user approval BEFORE running the full Test of Design. Shows the LLM-generated attributes, worksteps, and sample columns per control in a reviewable format. The user can then edit and approve them via the UI before proceeding.
  - preview_toe_attributes: Generate and preview TOE testing attributes for user approval BEFORE running the full Test of Effectiveness. Shows the LLM-generated attributes, worksteps, and sample columns per control in a reviewable format. The user can then edit and approve them via the UI before proceeding.
  - run_quality_comparison: Run a Quality Comparison: autonomously executes the full SOX audit pipeline (TOD with auto-approved schemas -> Sampling -> TOE), then compares our tool's results against a human auditor's pre-existing workpapers. Produces a comparison report identifying gaps in the human's work at three levels: per-control verdict, per-attribute, and per-sample. Requires a single project folder containing subfolders: RCM/ (with the RCM Excel), evidence/TOD/ and evidence/TOE/ (with Control ID subfolders), OutputWork/TOD/ (human's TOD workpaper Excel), and OutputWork/TOE/ (human's TOE workpaper Excel).
  - run_sampling_engine: Determine the number of samples required per control for the Test of Operating Effectiveness (TOE) using the KPMG sampling methodology or a custom sampling table. Uses each control's 'Control Frequency' and 'risk_level' columns to look up the required sample count, then writes the result into the 'count_of_samples' column. Run this BEFORE the TOE stage.
  - run_sox_scoping_engine: Run the SOX Scoping Engine (from folder 'sox_package 2') using dynamic input files. Accepts a trial balance file path and optional SOP/policy paths provided at runtime. No hardcoded input files are used.
  - run_test_of_design: Run Test of Design (TOD) — this is NOT Test of Effectiveness (TOE). TOD evaluates whether controls are adequately DESIGNED to mitigate risks (1 sample per control, PASS/FAIL). Use this when the user says 'TOD', 'test of design', or 'design test'. Do NOT use run_test_of_effectiveness or preview_toe_attributes for TOD.
  - run_test_of_effectiveness: Run Test of Operating Effectiveness (TOE) — this is NOT Test of Design (TOD). TOE evaluates whether controls OPERATED EFFECTIVELY in practice using multiple samples per control. Use this ONLY when the user says 'TOE', 'test of effectiveness', or 'effectiveness test'. Do NOT use this when the user asks for TOD/test of design — use run_test_of_design instead.
[DATA]
  - inspect_dataframe: Quick inspection of the current RCM DataFrame. Modes: 'info' (shape, dtypes, nulls), 'head'/'tail' (first/last N rows), 'describe' (statistics), 'columns' (list all columns), 'value_counts' (unique values for a column), 'sample' (random N rows), 'query' (filter with pandas query syntax), 'full' (return all rows, capped at 500 for context safety).
  - load_rcm: Load an RCM Excel or CSV file into the working state. Returns row count, columns, and a preview of the first 5 rows.
  - merge_suggestions: Merge user-selected AI suggestions into the RCM. Call after run_ai_suggestions and after the user has chosen which to keep.
  - modify_rcm: Modify the current RCM DataFrame. Actions: add_column, rename_column, update_values (case-insensitive conditional), delete_column, delete_rows (by condition or query), bulk_update (multiple columns at once), sort, filter_view (read-only filtered view).
  - remove_duplicates: Remove duplicate rows from the RCM based on user decisions. Call after run_deduplication and user review.
  - remove_failed_tod_controls: Remove controls that FAILED the Test of Design (TOD) from the RCM. This creates a clean RCM containing only PASS controls, which can then be used directly for Sampling and TOE. Optionally saves the removed controls to a separate file for audit trail.
[FILESYSTEM]
  - list_directory: List files and folders in a directory, optionally filtered by extension. Use recursive=true to walk subdirectories. Use this to find RCM files, evidence folders, or policy/SOP PDFs.
  - read_file: Read the contents of a file. Supports text files (.txt, .csv, .json, .py, .md, .log), Excel (.xlsx returns first sheet as table), and PDF (.pdf extracts text). Returns the first 200 lines by default. NOTE: Do NOT use this for RCM files — use load_rcm instead, which performs smart header detection, column mapping, and marker normalisation.
[UTILITY]
  - save_excel: Save the current RCM state to an Excel checkpoint file.
  - update_plan: Create or update your working plan. Use this to organise multi-step goals. Write markdown with checkboxes for tracking. The plan is YOUR scratchpad — the system will show it back to you each turn.
  - add_attribute: Add a new testing attribute to a control during the TOD/TOE preview stage. Provide the control ID, attribute name, description, and optionally the position (serial number) where it should be inserted. If no position is given, the attribute is appended at the end.
  - modify_attribute: Modify an existing testing attribute's name or description for a control during the TOD/TOE preview stage. Provide the control ID and attribute number (e.g. 1, 2, 3) along with the updated name and/or description.
  - remove_attribute: Remove a testing attribute from a control during the TOD/TOE preview stage. Provide the control ID and the attribute serial number to remove.
  - execute_python: Execute a Python code snippet for ad-hoc RCM data analysis. Pre-loaded variables: df (current RCM DataFrame), pd, np, os, json, math, datetime, re, openpyxl. Assign to 'result' to return structured data. If you reassign 'df', the RCM will be updated.
  - web_search: Search the web for SOX compliance, PCAOB standards, COSO framework, audit procedures, and general knowledge. Uses DuckDuckGo + LLM knowledge.

  16. UNIQUE CAPABILITIES OF THE NEW AUTONOMOUS AGENT

   1. TRUE AUTONOMY
      The LLM decides what to do at every turn. No hardcoded workflow.
  Ask 'just check for duplicates' → it loads RCM, runs dedup, shows results. DONE.
  Ask 'full audit' → it creates a plan, works through it, asks for input as needed.
  The same agent handles both. No mode switching. No pipeline enforcement.

   2. DYNAMIC PLANNING (update_plan tool)
      The agent writes its own plan as a markdown scratchpad.
  The plan is shown in every LLM context but never enforced by code.
  Agent can follow it, revise it, or completely deviate.
  Mid-course changes: 'actually skip the dedup' → agent updates plan and moves on.

   3. ANOMALY DETECTION (Reflector)
      Advisory notes are injected into tool results via _agent_notes.
  Catches: 100% TOD pass rates, zero TOE deviations, zero dedup pairs in large RCMs,
           very few or too many AI suggestions, <50% control assessment match.
  The LLM sees these notes and decides what to do — flag to user, re-examine, continue.

   4. 3-TIER MEMORY
      Episodic: Full conversation messages, windowed to last 40.
  Semantic: Auto-extracted key facts that survive context truncation.
    Example: 'RCM loaded: 28 rows, 15 columns' persists even after window slides.
  Plan scratchpad: The agent's working plan, always visible in context.

   5. TOOL AUTO-DISCOVERY
      Drop a .py file in agent/tools/ with a Tool subclass → it's registered on restart.
  No manual JSON schema editing. No handler mapping. Schema auto-generated.
  Example: add custom_analysis.py with class CustomTool(Tool) → available to LLM.

   6. EXECUTE_PYTHON SANDBOX
      The LLM can write and execute arbitrary Python code.
  Pre-loaded: df (current RCM), pd, np, os, json, math, datetime.
  If the code reassigns 'df', the RCM is automatically updated.
  Assign to 'result' to return structured data to the LLM.

   7. WEB SEARCH
      DuckDuckGo Instant Answer API + LLM knowledge fallback.
  The agent can look up SOX regulations, PCAOB standards, COSO framework on the fly.
  No pre-loaded knowledge required — real-time information retrieval.

   8. INSPECT_DATAFRAME (8 modes)
      info: shape, dtypes, nulls, memory usage
  head/tail: first/last N rows
  describe: full statistics
  columns: all columns with dtype, non-null count, unique count
  value_counts: unique values for any column
  sample: random N rows
